Ниже только те договоренности, которые применимы как общие frontend-правила и архитектурный стиль.

## 1) Архитектурные договоренности

- Использовать FSD-подобное разбиение: `app`, `pages`, `widgets`, `features`, `entities`, `shared`.
- Ограничивать ответственность модулей:
  - `app` — инициализация приложения, глобальные стили и корневой роут.
  - `pages` — сборка экранов из виджетов и фич.
  - `widgets` — крупные UI-блоки страницы.
  - `features` — пользовательские сценарии, переключатели, формы, контроллеры.
  - `entities` — доменная модель, типы, вычисления и селекторы.
  - `shared` — переиспользуемые утилиты, UI-примитивы, константы.
- Для каждого публичного блока по возможности использовать `index.ts` как публичный API.

## 2) Импорты

- Предпочитать абсолютные импорты по алиасу (в kormos-ui используется `@/`).
- Избегать глубоких относительных импортов вида `../../../`.
- Импортировать из публичного API слоя/слайса, а не из внутренних файлов, если это возможно.

## 3) Правила по React-компонентам

- Один компонент верхнего уровня на файл.
- Не объявлять React-компоненты внутри других компонентов.
- Не использовать inline-функции в JSX без необходимости.
- Держать компоненты маленькими и выносить логику в отдельные функции/хуки.

## 4) TypeScript и качество кода

- Строгая типизация входов/выходов публичных функций.
- Не оставлять неиспользуемые переменные и аргументы (допускается `_name` для намеренно неиспользуемых).
- Константы и «магические числа» выносить в именованные значения там, где это улучшает читаемость.
- Явно обрабатывать ошибочные ветки пользовательских сценариев.

## 5) UI и стили

- Для переиспользуемых UI-компонентов не пробрасывать случайные внешние стили, ломающие дизайн-контракт.
- Расширение визуальных вариантов делать через API компонента, а не хаотичные переопределения.
- Поддерживать единообразие размеров, отступов и типографики между однотипными блоками.

## 6) Lint-ориентиры (из kormos-ui)

- Базовый стек: ESLint + TypeScript + React Hooks + React Refresh.
- Важные ограничения:
  - запрет вложенных компонентов;
  - контроль неиспользуемых переменных;
  - аккуратная политика по magic numbers;
  - строгие проверки hooks.

## 7) Неприменимые для этого проекта правила из kormos-ui

Ниже правила остаются в исходном проекте и не переносятся 1:1 сюда:
- Solana/Anchor-специфика, PDA, refresh, rust_decimal.
- Протокольные формулы и ограничения DeFi-домена.

Они сохранены в `kormos-ui/FRONTEND_CONTEXT.md` и нужны только для домена Korm.
